# Team Roles

## Team Structure

Small, lean, cross-functional team. All core roles exist; individuals may cover multiple. No dedicated project management layer — communication is direct, process is minimal.

| Role             | File          | Owns                                                        |
| ---------------- | ------------- | ----------------------------------------------------------- |
| Product Owner    | `po.md`       | Vision, backlog, delivery alignment                         |
| Tech Lead        | `techlead.md` | Architecture, technical direction                           |
| Engineer         | `eng.md`      | Design, build, maintain, UI implementation                  |
| QA               | `qa.md`       | Quality, acceptance criteria                                |
| Business Analyst | `ba.md`       | Requirements, basic UI/UX flows, bridging business and tech |

Escalation paths are short. Most decisions are made at the role level. Cross-role decisions go to the Tech Lead.

---

## Role Definition Principles

These principles apply to every role file in this directory.

- **Action-oriented.** Use verb-first, imperative language. Write instructions, not descriptions.
- **Explicit boundaries.** State clearly what the role decides autonomously and when to escalate.
- **Inputs and outputs.** Define what the role consumes and produces, not internal steps.
- **Principles over rules.** Use judgment heuristics that generalize, not exhaustive rule lists.
- **Lean-calibrated.** Omit anything that only matters at 10+ people. Keep it relevant to this team.
- **No redundancy.** If something is implied by context or another principle, leave it out.
